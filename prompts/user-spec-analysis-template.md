# User Prompt Template — Single Spec Analysis

## Runtime Interpolation Template

This file is the runtime user prompt for the single-spec ambiguity analysis call. Before being sent to the AI model, the workflow replaces all `{{placeholder}}` values with actual values from the normalized spec input.

**Fields populated by the workflow at runtime:**

| Placeholder | Source | Notes |
|---|---|---|
| `{{analysis_mode}}` | Webhook input `analysis_mode` field | Always `single_spec` for this prompt |
| `{{document_title}}` | Spec object `title` field | The spec's title as provided |
| `{{document_type}}` | Spec object `type` field | e.g., `prd`, `technical_spec`, `user_stories`, `implementation_brief`, `feature_request` |
| `{{source_summary}}` | Generated by Normalize node | One sentence describing detected sections |
| `{{word_count}}` | Generated by Extract Signals node | Approximate word count of the raw text |
| `{{section_count}}` | Generated by Extract Signals node | Number of named sections detected |
| `{{preferred_tone}}` | Webhook input `preferred_tone` field | `direct`, `explanatory`, or `minimal` |
| `{{document_text}}` | Spec object `raw_text` field | Full normalized spec text |
| `{{known_sections}}` | Generated by Extract Signals node | Comma-separated list of detected section names |
| `{{detected_open_questions}}` | Generated by Extract Signals node | Any open question text detected deterministically, or "none detected" |
| `{{detected_out_of_scope_items}}` | Generated by Extract Signals node | Any out-of-scope text detected deterministically, or "none detected" |

---

## Allowed Category Identifiers (Reference)

The `ambiguity_categories` array in the output must use only these exact string values:

`vague_language` · `missing_acceptance_criteria` · `undefined_ownership` · `missing_decision_point` · `scope_boundary_gap` · `dependency_blind_spot` · `untestable_requirement` · `contradictory_requirement` · `assumption_without_validation` · `low_signal_noise`

Do not use synonyms or alternative phrasings. The system prompt defines the full meaning of each identifier.

## Template

```
Analyze the following document for ambiguity patterns.

Analysis mode: {{analysis_mode}}
Document title: {{document_title}}
Document type: {{document_type}}
Source summary: {{source_summary}}
Approximate word count: {{word_count}}
Sections detected: {{section_count}}
Preferred output tone: {{preferred_tone}}

---

STRUCTURAL SIGNALS (pre-detected by the system before this call):

Sections found in document: {{known_sections}}
Open questions already present in document: {{detected_open_questions}}
Out-of-scope items already present in document: {{detected_out_of_scope_items}}

Use these signals as context. If open questions are already explicitly listed, note that. If out-of-scope items are already listed, note that. Do not penalize a document for problems it has already acknowledged. But do flag gaps that remain unresolved even in the presence of acknowledged questions.

---

DOCUMENT TEXT:

{{document_text}}

---

ANALYSIS INSTRUCTIONS:

1. Read the full document text above.
2. Identify every ambiguity pattern present from the allowed category list in your system instructions.
3. For each category you identify, find specific evidence from the document text. Do not flag a category if you cannot point to specific language that supports it.
4. Populate all required output fields. Return empty arrays where no items apply — do not omit fields.
5. Set severity level according to the severity rules in your system instructions.
6. Set confidence level according to your certainty about the diagnosis, not the document quality.
7. Recommendations must be specific, actionable, and tied to a named problem. Never write "improve clarity." Write what specifically needs to be defined, assigned, or stated, and where in the document it should appear.

---

EXPECTED OUTPUT FORMAT:

Respond with a single, valid JSON object matching this schema exactly. No markdown, no code fences, no prose before or after the JSON.

{
  "ambiguity_summary": "2–5 sentences. Describe which ambiguity patterns are present and why they create delivery risk. Do not summarize the document content.",
  "ambiguity_categories": ["list of category strings from the allowed set"],
  "severity_level": "none | low | medium | high | critical",
  "highest_risk_sections": ["section names from the document"],
  "vague_terms_detected": ["exact or near-exact terms from the document text"],
  "missing_decisions": ["each is a specific unresolved decision implied by the document"],
  "undefined_ownership_flags": ["each names a specific ownership gap with the relevant text"],
  "non_testable_requirements": ["each is a specific requirement that cannot be verified as stated"],
  "dependency_gaps": ["each names a specific dependency implied but not explicitly stated"],
  "scope_boundary_gaps": ["each is a specific scope boundary that is missing or undefined"],
  "contradictions_detected": ["each describes a contradiction as a pair of conflicting statements from the document"],
  "assumptions_needing_validation": ["each is a specific assumption presented as fact that has not been validated"],
  "recommended_clarifications": ["each is a specific, actionable item tied to an identified ambiguity pattern"],
  "recommended_next_step": "single sentence — the most important action before this spec proceeds",
  "confidence_band": "low | medium | high",
  "needs_human_review": true | false,
  "review_reason": "string explaining why human review is needed, or null if needs_human_review is false"
}
```
