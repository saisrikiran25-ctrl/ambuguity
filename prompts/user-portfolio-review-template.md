# User Prompt Template — Portfolio Review

## Runtime Interpolation Template

This file is the runtime user prompt for the portfolio-level review call. The workflow populates all `{{placeholder}}` values before sending the prompt to the AI model.

**Fields populated by the workflow at runtime:**

| Placeholder | Source | Notes |
|---|---|---|
| `{{spec_count}}` | Count of specs in the portfolio | Total number of specs being reviewed |
| `{{preferred_tone}}` | Webhook input `preferred_tone` field | `direct`, `explanatory`, or `minimal` |
| `{{portfolio_date_range}}` | Webhook input `time_window_start` and `time_window_end` | ISO 8601 date range, or "not specified" |
| `{{spec_summaries}}` | Generated by Prep Portfolio Review node | JSON array of individual spec diagnostic results |
| `{{high_risk_count}}` | Generated by Prep Portfolio Review node | Count of specs with severity high or critical |
| `{{needs_review_count}}` | Generated by Prep Portfolio Review node | Count of specs with needs_human_review true |
| `{{repeated_category_list}}` | Generated by Prep Portfolio Review node | Comma-separated list of categories appearing in 2+ specs |

---

## Allowed Category Identifiers (Reference)

When populating `repeated_ambiguity_categories.category`, use only these exact string values:

`vague_language` · `missing_acceptance_criteria` · `undefined_ownership` · `missing_decision_point` · `scope_boundary_gap` · `dependency_blind_spot` · `untestable_requirement` · `contradictory_requirement` · `assumption_without_validation` · `low_signal_noise`

These are the same ten canonical identifiers used in individual spec analysis. Do not use synonyms. Do not invent new categories. The system prompt defines the full meaning of each.

## Template


```
Conduct a portfolio-level review of the following spec analyses.

Portfolio size: {{spec_count}} specs
Date range: {{portfolio_date_range}}
Preferred output tone: {{preferred_tone}}
Specs with high or critical severity: {{high_risk_count}}
Specs flagged for human review: {{needs_review_count}}
Categories appearing in multiple specs: {{repeated_category_list}}

---

INDIVIDUAL SPEC DIAGNOSTIC RESULTS:

{{spec_summaries}}

---

PORTFOLIO REVIEW INSTRUCTIONS:

1. Read all individual spec diagnostic results above.
2. Identify patterns that recur across two or more specs. Only name a pattern as "recurring" if it appears in at least two specs.
3. Identify which specs represent the highest delivery risk if they proceed to engineering without clarification.
4. Identify which specs have systemic ambiguity severe enough to require a full rewrite rather than targeted clarifications.
5. Produce portfolio-level recommendations — actions the team should take at a process or team level, not spec-by-spec fixes. Spec-by-spec fixes are already in the individual analyses.
6. Set confidence based on the quality of the individual analyses you received as input. If multiple individual analyses have low confidence, your portfolio conclusions should also be low confidence.

---

WHAT TO LOOK FOR:

Recurring ownership gaps:
- Is the same function (e.g., backend delivery, data compliance, notification architecture) consistently unowned across specs?
- Are ownership gaps isolated to one spec, or distributed across the set?

Repeated missing acceptance criteria:
- Do multiple specs describe requirements without measurable success conditions?
- If so, this likely indicates a template or process gap, not just individual authoring failure.

Weak scope discipline:
- Are out-of-scope sections consistently missing?
- Are assumptions consistently unvalidated?
- Are open questions consistently not listed?

Common missing decisions:
- Are the same types of decisions being deferred across multiple specs?
- Example: If three specs all defer notification delivery mechanism decisions, that is a portfolio-level gap.

Specs at highest delivery risk:
- Which specs, if they proceeded to engineering breakdown today, would most likely result in misaligned builds?
- Consider: severity level, number of ambiguity categories, presence of contradictions, critical ownership gaps.

---

EXPECTED OUTPUT FORMAT:

Respond with a single, valid JSON object. No markdown, no code fences, no prose before or after the JSON.

{
  "review_summary": "3–6 sentences on overall spec quality, most significant patterns, and portfolio-level delivery risk",
  "top_risk_specs": [
    {
      "title": "exact spec title from the input",
      "risk_reason": "one sentence on why this spec represents highest delivery risk"
    }
  ],
  "repeated_ambiguity_categories": [
    {
      "category": "category identifier",
      "occurrence_count": 0,
      "pattern_description": "one sentence on how this category manifests across specs"
    }
  ],
  "common_decision_gaps": ["each is a type of decision consistently deferred across multiple specs"],
  "common_ownership_gaps": ["each is a type of ownership consistently undefined across multiple specs"],
  "common_acceptance_criteria_failures": ["each is a type of requirement that consistently lacks acceptance criteria"],
  "specs_needing_rewrite": [
    {
      "title": "exact spec title",
      "reason": "one sentence on why targeted clarification is insufficient"
    }
  ],
  "portfolio_recommendations": ["each is a specific, portfolio-level action — not a spec-level fix. Focus on process, template, or team-level improvements."],
  "confidence_band": "low | medium | high",
  "needs_human_review": true | false,
  "review_reason": "string if needs_human_review is true, otherwise null"
}
```
